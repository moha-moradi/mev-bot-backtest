# MEV Bot Backtest Engine - Implementation Plan

## Project Overview
Build a high-fidelity historical backtest engine for MEV strategies on EVM-compatible chains using Rust. The engine will replay historical blocks and simulate MEV bot profits across five strategy phases.

## Technology Stack
- Language: Rust
- EVM Execution: revm
- Async Runtime: Tokio
- RPC/Data: alloy-rs
- State Persistence: RocksDB
- CLI: clap v4
- Config: TOML
- Reporting: comfy-table + CSV/JSON export

## Implementation Plan by Phase

### Phase 2: Data Ingestion & Caching Layer
**Goal**: Reliable block data fetching, storage, and day-based range resolution via RPC.
**Steps**:
1. Add dependencies: alloy-rs, rocksdb, bincode/postcard, indicatif
2. Define internal data types:
   - BlockData (number, hash, timestamp, base_fee_per_gas, gas_limit, miner)
   - TxData (hash, index, from, to, input, value, gas_limit, max_fee, max_priority_fee, nonce, access_list)
   - ReceiptData (tx_hash, tx_index, status, gas_used, cumulative_gas_used, logs)
   - LogData (address, topics[4], data)
3. Implement RpcClient wrapper around alloy-rs with retry logic (exponential backoff, max 5 retries):
   - get_block_number(), get_block(), get_block_timestamp(), get_receipts()
   - get_storage_at(), get_account(), get_code()
4. Implement block range resolver:
   - --days N: binary search on timestamps using get_block_timestamp()
   - --blocks N: calculate from tip
   - --block NUMBER: single block
   - --from-block/--to-block: direct use
   - Cache resolved range in RunManifest
5. Implement CacheStore with RocksDB:
   - Key schema: block:{chain_id}:{num}, txs:{chain_id}:{num}, etc.
   - Values serialized with bincode/postcard
6. Implement RunManifest storage:
   - Fields: run_id, chain, start_block, end_block, resolved_at, range_mode, strategies, flash_loan_provider
7. Implement fetch subcommand:
   - Resolve range, iterate blocks with progress bar
   - Cache-check, fetch block/tx/receipt data, store to cache
   - Print summary: fetched/cached counts, elapsed time
8. Implement parallel fetching with Tokio semaphore (limit: min(parallelism, 20))
9. Implement cache integrity check: verify block+txs+receipts present, report gaps
10. Implement lazy account/slot fetcher for revm (fetch at block N-1, cache, return)

### Phase 3: EVM State Replay Engine
**Goal**: Replay historical blocks exactly using revm for mid-block state snapshots.
**Steps**:
1. Add dependency: revm (full EVM features)
2. Implement CachedRpcDb (implements revm Database trait):
   - basic(address): check cache, then lazy-fetch via RPC client
   - code_by_hash(): same pattern
   - storage(address, slot): same pattern
   - block_hash(number): fetch from cache
3. Implement BlockReplayer:
   - Input: block number, CachedRpcDb
   - Method: replay_to(tx_index) -> Result<(CacheDB, Vec<ExecutionResult>)>
   - Load BlockData and TxData from cache
   - Configure revm Environment (block.number, timestamp, coinbase, gas_limit, basefee, chain_id, spec_id)
   - For each tx[0..=tx_index]: set tx env, call evm.transact_commit(), store ExecutionResult
   - Return final CacheDB state + results
4. Implement receipt verification:
   - Compare ExecutionResult vs ReceiptData: status, gas_used, log count, topics[0]
   - Log WARN for mismatches
5. Implement StateSnapshot.fork(): clone cache maps for independent state
6. Implement replay subcommand (debug):
   - mev-backtest replay --block <N> [--tx-index <I>]
   - Print per-tx results and receipt match summary
   - Reject --days/--blocks flags (specific block only)
7. Implement SpecId selection per chain/block:
   - Map (chain_id, block_number) -> revm SpecId (LONDON, PARIS, etc.)

### Phase 4: DEX Price Graph & Event Decoder
**Goal**: Parse swap events from logs, maintain live price graph reflecting pool state.
**Steps**:
1. Define PriceGraph:
   - Nodes: token addresses (checksummed)
   - Edges: Pool {address, dex_type, dex_version, token_a, token_b, reserve_a/reserve_b (V2) or sqrt_price_x96/tick/liquidity (V3), fee_bps, tick_spacing, is_concentrated_liquidity}
2. Implement log decoders per DEX:
   - AMM V2 (Uniswap V2 clones): Sync(reserve0, reserve1)
   - Concentrated Liquidity V3 (Uniswap V3 clones): Swap(..., sqrtPriceX96, liquidity, tick), Mint/Burn(..., tickLower, tickUpper, amount, ...)
   - Other AMM: Curve StableSwap (TokenExchange), Balancer V2 (Swap)
3. Implement PriceGraph.apply_update(PoolUpdate):
   - V2: update reserves from Sync
   - V3: update sqrt_price_x96, tick, liquidity from Swap/Mint/Burn
   - Curve/Balancer: update virtual balances
4. Implement pool registry loader from pools.json per chain:
   - Schema: {address, dex_type, dex_version, is_concentrated_liquidity, token0, token1, fee_bps, tick_spacing (V3), deploy_block}
   - Minimum pools for Polygon: QuickSwap V2 (top 100), SushiSwap V2 (top 50), QuickSwap V3 (top 100), Uniswap V3 (top 50), SushiSwap V3 (top 50), Curve (top 20), Balancer V2 (top 20)
5. Implement PriceGraph.init_from_block(block_number):
   - Fetch reserves/state at block-1 for registry pools, populate initial edges
6. Implement quote functions:
   - quote_v2: constant-product formula with fee deduction
   - quote_v3: V3 tick math via Uniswap V3 library port
   - quote_path: chain quotes through mixed V2/V3 hops
7. Add test: replay known block, apply swap events, compare computed reserves/sqrtPriceX96 against eth_call results (target: <0.01% mismatch)

### Phase 5: Strategy A - Two-Hop Arbitrage
**Goal**: Detect/simulate buy token A→B on pool X, sell B→A on pool Y via flash loan.
**Steps**:
1. Implement TwoHopArbDetector:
   - After each tx updates PriceGraph, for each pool pair sharing (token_a, token_b):
   - Compute optimal input: optimal_in = (sqrt(r_a0*r_b0*r_a1*r_b1) - r_a0*r_b0) / (r_a0 + r_a1) (fee-adjusted)
   - If optimal_in > min_input_usd: emit Candidate
2. Implement FlashLoanWrapper:
   - Input: token, amount, chain config, forced_provider
   - If forced_provider != auto: check only that provider, return fee or SKIP
   - If forced_provider == auto: priority chain:
     1. Balancer V2 Vault balance >= amount → fee=0
     2. Aave V3 liquidity >= amount → fee = amount * 0.0005
     3. Uniswap deepest pool → fee = amount * pool_fee_bps
     4. SKIPPED_NO_FLASHLOAN
   - Returns SelectedProvider {provider, fee_amount}
3. Implement TwoHopArbSimulator:
   - a. Call FlashLoanWrapper for input token
   - b. Build simulated tx: flash loan → swap X → swap Y → repay
   - c. Execute in revm forked state
   - d. Binary-search optimal input (≤10 iterations)
   - e. Compute:
        gross = amount_out - amount_in
        gas_cost = gas_used * effective_gas_price
        bribe = gross * coinbase_bribe_pct / 100
        net = gross - gas_cost - bribe - flash_loan_fee
   - f. Return ArbResult {block, tx_index_after, pool_a, pool_b, token_in, amount_in, amount_out, flash_loan_provider, flash_loan_fee, gross_profit, gas_used, gas_cost, coinbase_bribe, net_profit}
4. Integrate into block replay loop: after each tx and PriceGraph update, call detector then simulator; store results with net_profit > 0
5. Implement USD conversion: token price map from bundled historical prices CSV (daily snapshots)
6. Implement Phase 5 report output:
   - Two-Hop Arbitrage Results table with metrics
   - Flash loan provider breakdown (Balancer V2, Aave V3, Uniswap Flash Swap)

### Phase 6: Strategy B - Multi-Hop Arbitrage
**Goal**: Detect 3-4 hop arbitrage cycles across any pool types.
**Steps**:
1. Implement Bellman-Ford cycle detector on PriceGraph:
   - Edge weights: -log(quote_output / quote_input)
   - Detect negative-weight cycles up to max_hops (default 4)
   - Record cycles as Vec<(pool, token_in, token_out)>
2. Implement MultiHopArbSimulator:
   - Same structure as TwoHopArbSimulator:
     - Calls FlashLoanWrapper (respects forced provider)
     - Chains N swaps through any mix of V2/V3 pools
     - Binary-searches optimal input
     - Returns MultiHopArbResult with path field
3. Implement deduplication: if 3-hop cycle contains 2-hop sub-cycle already found, keep only more profitable
4. Add multi_hop_arb to CLI strategies; runnable independently
5. Extend report with per-path breakdown table and combined two_hop + multi_hop totals row

### Phase 7: Strategy C - JIT Liquidity
**Goal**: Simulate adding concentrated liquidity before large swap, earning fees, then removing.
**Steps**:
1. Guard at initialization: filter pool registry to CL pools only. If none for chain:
   - Print: "No concentrated liquidity pools found for chain <name>. JIT strategy requires Uniswap V3 fork pools. Skipping."
2. Implement LargeSwapDetector (pre-scan per block):
   - Decode V3 router calldata only
   - Flag swaps above min_swap_usd (default 50,000)
   - Candidate's pool must have is_concentrated_liquidity = true
3. Implement JitSimulator per candidate at tx_index I:
   - a. Fork state at tx_index I - 1
   - b. Compute tick range:
        tick_lower = floor(tick / spacing) * spacing - spacing
        tick_upper = tick_lower + (2 * spacing)
        Clamp to [MIN_TICK, MAX_TICK]
   - c. Simulate mint tx
   - d. Simulate victim tx; record fees earned
   - e. Simulate burn tx; record tokens received
   - f. Compute:
        fees_usd = fees_token0_usd + fees_token1_usd
        il_usd = deposited_usd - received_usd (pre-fee)
        gas_cost = (gas_mint + gas_burn) * gas_price
        net = fees_usd - il_usd - gas_cost
   - g. Return JitResult (includes il_usd, no flash loan fields)
4. Implement JIT report:
   - JIT Liquidity Results table (Concentrated Liquidity Only)
   - Metrics: eligible pools, victims targeted, net-positive, total fees earned, total IL, total gas cost, total net profit, avg per opportunity

### Phase 8: Strategy D - JIT + Arb Combo
**Goal**: After JIT burn, detect/capture arb from token imbalance (arb leg any pool type).
**Steps**:
1. After each successful JIT simulation:
   - Run Bellman-Ford cycle detector (Phase 6) on post-burn forked state
2. If arb cycle found:
   - Call FlashLoanWrapper for arb input token (respects --flash-loan-provider)
   - Simulate arb on same forked state
3. Compute combined net profit:
   - arb_flash_fee = FlashLoanWrapper fee for arb token
   - combined_net = jit_net + arb_gross - arb_gas_increment - arb_flash_fee
4. Return JitArbResult extending JitResult with:
   - arb_path, arb_gross, arb_flash_provider, arb_flash_fee, arb_gas_increment, combined_net_usd
5. Implement report comparison:
   - JIT vs JIT+Arb Comparison table
   - Metrics: JIT alone net, JIT+Arb net, Arb uplift, Uplift %, Arb flash fees

### Phase 9: Strategy E - Sandwich Arbitrage
**Goal**: Detect large victim swaps, simulate frontrun+victim+backrun, compute net profit.
**Steps**:
1. Implement SandwichDetector (pre-scan per block):
   - Decode V2 and V3 router calldata
   - Flag txs with estimated price impact >= min_slippage_pct (default 0.5%)
   - Emit SandwichCandidate with pool_type field
2. Implement SandwichSimulator per candidate at tx_index I:
   - a. Fork state at I - 1
   - b. Call FlashLoanWrapper for frontrun input token (respects --flash-loan-provider)
   - c. Binary-search optimal frontrun amount:
        simulate frontrun (correct call for pool_type)
        → simulate victim (must succeed in revm)
        → simulate backrun
        → compute net = backrun_out - frontrun_in - flash_fee - gas_cost - bribe
   - d. Constraint: victim tx must not revert
   - e. Return SandwichResult with flash_loan_provider, flash_loan_fee fields
3. Implement Sandwich report:
   - Sandwich Arbitrage Results table
   - Metrics: victims scanned, viable candidates, net-positive, skipped (no loan), total gross, total flash fees, total gas cost, total net

## Known Limitations (to document in README)
- Does not model competing bots (real profits lower)
- Does not model mempool latency or block builder negotiation
- JIT and JIT+Arb only supported on Uniswap V3 fork pools
- Flash loan availability checked via pre-block balance (intra-block changes not modeled)
- Forced flash loan provider: opportunities lost if provider lacks liquidity (counted as SKIPPED_NO_FLASHLOAN)
- --days N uses binary search on block timestamps (depends on archive node accuracy)
- Token prices use daily snapshots (USD values approximate)
- Capital cost for JIT positions not modeled
- Non-standard EVM opcodes may require revm patches
